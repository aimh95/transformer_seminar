학습 자료 · Transformer Deep Dive

Transformer를 "연산 그래프"로 이해하는 방식

개념 암기가 아니라 Tensor Flow, Shape, Mask, Gradient로 납득하기

"Attention은 관계를 본다"는 설명은 틀리지 않지만, 그것만으로는 부족하다. 이 자료는 왜 Q/K/V가 그런 역할로 학습되는지, Mask가 정확히 어디에 적용되는지, FFN이 Attention과 어떻게 다른지, Residual이 왜 shape 일치를 강제하는지를 실제 tensor shape과 연산 순서로 납득하는 방식을 설명한다.

1

일반적인 이해 vs 내가 원하는 이해

일반적인 개념 설명

  • Transformer는 Attention을 사용한다.
  • Query는 찾는 정보다.
  • Key는 가진 정보의 특징이다.
  • Value는 전달할 정보다.
  • FFN은 비선형성을 추가한다.
  • Residual connection은 gradient를 안정화한다.
  • Mask는 미래 token을 가린다.

→ 개념적으로 틀리지 않지만, 실제로 "그게 성립하는지" 연산 그래프에서 확인할 수 없다.

내가 원하는 이해

  • Q, K, V는 어떤 X와 W에서 만들어지는가?
  • QKᵀ의 shape은 어떻게 되는가? row와 column은 무엇을 의미하는가?
  • Mask는 Q/K에 적용되는가, Score Matrix에 적용되는가?
  • Loss가 줄어들 때 W^Q, W^K, W^V는 어떤 방향으로 업데이트되는가?
  • FFN은 token 간 정보를 섞는가, 각 token을 독립적으로 변환하는가?
  • Residual이 왜 d_model로 shape을 강제하는가?
개념을 믿기 전에, 실제 연산 그래프에서 그 말이 성립하는지 확인하려는 방식
2

Transformer 전체 구조를 연산 흐름으로 보기

Transformer를 "멋진 아이디어 모음"이 아니라, 입력 tensor가 여러 weight matrix, mask, softmax, residual connection을 지나며 변환되는 연산 그래프로 본다.

Redrawn based on "Attention Is All You Need", Vaswani et al., 2017.

Encoder self-attention

  • 모든 source token끼리 서로를 참조 가능
  • Padding mask를 사용 (look-ahead mask ✗)
  • Q, K, V 모두 같은 Encoder hidden state X에서 생성
  • Output: H_enc ∈ ℝ^{T × d_model}

Decoder의 세 가지 sublayer

  • Masked Self-Attention: Look-ahead mask 적용, 현재까지의 token만 참조
  • Cross-Attention: Q는 Decoder state, K/V는 H_enc — encoder 정보를 읽는다
  • FFN: 각 token을 독립적으로 변환 (token 간 섞음 없음)
Multi-Head Attention 주의: 헤드가 sequence를 나누는 것이 아니라, d_model 차원을 h개의 헤드로 분할하여 각각 독립적으로 attention을 수행한다. d_k = d_model / h (논문 기준 d_model=512, h=8 → d_k=64).
3

왜 Q/K/V 역할을 그냥 외우면 부족한가?

"Query가 찾는 쪽이고 Key가 찾아지는 쪽"이라는 말이 왜 성립하는지는 실제 행렬 shape과 score matrix 구조를 보면 자연스럽게 이해된다.

Redrawn based on "Attention Is All You Need", Vaswani et al., 2017.

# X: 입력 embedding (모든 token의 representation) X ∈ ℝ^{seq_len × d_model} # 예: [<SOS>, Le, chat, est] → (4, 512) Q = X · W^Q # W^Q ∈ ℝ^{d_model × d_k} → Q ∈ ℝ^{seq_len × d_k} K = X · W^K # W^K ∈ ℝ^{d_model × d_k} → K ∈ ℝ^{seq_len × d_k} V = X · W^V # W^V ∈ ℝ^{d_model × d_v} → V ∈ ℝ^{seq_len × d_v} Score = Q · Kᵀ # (seq_len × d_k) × (d_k × seq_len) = (seq_len × seq_len) # score[i][j] = q_i · k_j ← i번째 Query와 j번째 Key의 유사도 AttWeight = softmax(Score / √d_k) # √d_k: dot product 분산 보정 (seq_len이 아님) Output = AttWeight · V # (seq_len × seq_len) × (seq_len × d_v) = (seq_len × d_v)

Query — 찾는 쪽인 이유

Q가 Score Matrix의 row를 만들기 때문이다. i번째 row는 i번째 위치가 어떤 token들을 얼마나 참고할지 결정한다. 즉 "내가 무엇을 찾고 있는가"에 해당하는 표현이 row 방향으로 작용한다.

Key — 찾아지는 쪽인 이유

K가 Score Matrix의 column 방향에서 여러 Query와 비교되기 때문이다. j번째 Key는 여러 Query에게 "내가 지금 참고할 만한 token인가?"라는 방식으로 평가된다.

Q와 K가 처음부터 의미를 가지는 것은 아니다.
학습 초기에는 W^Q, W^K, W^V가 랜덤이기 때문에 Query, Key, Value에도 특별한 의미가 없다. 하지만 loss를 줄이는 과정에서 attention score가 정답 예측에 유리한 방향으로 조정되면서, Query는 현재 위치가 필요한 정보를 찾는 표현으로, Key는 참고 대상 token이 검색될 수 있는 특징으로 해석 가능해진다. 역할은 학습되는 것이지, 설계되는 것이 아니다.
4

Mask가 들어가면 행렬 연산이 어떻게 바뀌는가?

핵심: Mask는 Q와 K 자체를 가리는 것이 아니라, QKᵀ로 만든 Attention Score Matrix의 특정 위치를 −∞로 막는다. −∞가 들어간 위치는 softmax 이후 ≈ 0의 attention weight가 되어, Decoder는 미래 token을 사실상 참고할 수 없다.

Redrawn based on "Attention Is All You Need", Vaswani et al., 2017.

아래 표는 인터랙티브 버전이다. 어두운 셀(−∞)에 마우스를 올리면 설명이 표시된다.

<SOS> Le chat est
<SOS> 0 attend 가능 −∞ 차단됨 −∞ 차단됨 −∞ 차단됨
Le 0 attend 가능 0 attend 가능 −∞ 차단됨 −∞ 차단됨
chat 0 attend 가능 0 attend 가능 0 attend 가능 −∞ 차단됨
est 0 attend 가능 0 attend 가능 0 attend 가능 0 attend 가능
# Masked Attention 계산 순서 1. Q = X · W^Q 2. K = X · W^K 3. V = X · W^V 4. Score = Q · Kᵀ # (seq_len × seq_len) 5. MaskedScore = Score + Mask # Mask ∈ {0, −∞} 6. AttWeight = softmax(MaskedScore / √d_k) 7. Output = AttWeight · V

Encoder에서 사용하는 Mask

Encoder self-attention에는 look-ahead mask가 없다. Source sequence에서는 모든 위치끼리 자유롭게 참조 가능하다. 대신 padding 위치(실제 token이 없는 위치)를 막는 padding mask만 사용한다.

Decoder에서 사용하는 Mask

Decoder masked self-attention에는 look-ahead mask를 사용한다. 학습 시 전체 target sequence를 한번에 입력하지만, 각 위치는 자신 이전의 token만 볼 수 있다. Cross-Attention에서는 K/V가 Encoder 전체에서 오므로 별도 look-ahead mask가 없다.

FFN은 Attention과 무엇이 다른가?

Redrawn based on "Attention Is All You Need", Vaswani et al., 2017.

Attention의 역할

"어떤 token을 참고할지"를 결정한다. Score Matrix = QKᵀ 이 token 간 관계를 계산하므로, h₀ = f(x₀, x₁, ..., x_{T−1}) 형태로 모든 token의 정보를 혼합한다.

shape: (seq_len, d_model) → (seq_len, d_model) Score: ℝ^{seq_len × seq_len} ← token 간 관계

FFN의 역할

"각 token representation을 어떻게 더 유용한 feature로 변환할지"를 학습한다. token 간 정보를 새로 섞지 않는다. 각 token position에 동일한 MLP를 독립적으로 적용한다.

FFN(x) = max(0, x·W₁ + b₁) · W₂ + b₂ d_model → d_ff → d_model 512 → 2048 → 512 # 논문 기준 4×
FFN이 마지막에 다시 d_model로 돌아오는 이유: Residual connection 때문이다.
Output = LayerNorm(x + Sublayer(x))
입력 x와 sublayer output을 더하려면 shape이 같아야 한다. FFN 내부에서 d_model → d_ff(=2048)로 확장 후 다시 d_model(=512)로 압축하는 구조는 이 제약에서 기인한다.
6

이 질문 방식의 실무적 장점

장점 1
Shape mismatch 디버깅에 강함

tensor shape을 항상 추적하기 때문에 런타임 에러 원인을 빠르게 찾는다.

예: Attention output (seq_len, d_model)이 FFN 입력으로 들어갈 때 shape 불일치가 발생하면 residual connection에서 더하는 단계임을 바로 알 수 있다.
장점 2
Mask broadcasting 오류 추적 용이

Mask shape과 Score Matrix shape의 broadcasting 규칙을 이해하고 있어서 shape 불일치나 잘못된 차원 적용을 감지할 수 있다.

예: causal mask shape이 (seq_len, seq_len)인지, batch/head dimension으로 broadcast되어야 하는지 (batch, num_heads, seq_len, seq_len)인지 확인 가능.
장점 3
논문 수식을 코드로 옮기기 쉬움

수식의 각 항이 어떤 tensor 연산인지 직접 매핑할 수 있다.

예: Attention(Q,K,V) = softmax(QKᵀ/√d_k)V → torch.softmax(Q @ K.T / math.sqrt(d_k), dim=-1) @ V 를 직접 쓸 수 있다.
장점 4
ONNX / TFLite / ExecuTorch / CoreML 변환 분석

graph export 시 어떤 op이 변환되는지, 어디서 shape 변환이 발생하는지 이해한다.

예: softmax 위치, reshape op, transpose 순서가 변환 후 달라지면 어디서 문제가 생겼는지 graph 기준으로 분석.
장점 5
KV cache / quantization / prefill-decode 이해

inference 최적화 주제들이 결국 tensor shape, cache 전략, 연산 재사용에서 나온다는 것을 안다.

예: KV cache는 매 decode step마다 K, V를 재계산하지 않고 이전 결과를 축적해 (seq_len, d_k) → (cache_len, d_k)로 재사용.
장점 6
겉핥기 설명에 속지 않음

"Attention이 관계를 본다"는 말이 Score Matrix의 shape과 연산 방식으로 성립하는지 직접 확인할 수 있다.

예: "FFN이 token 간 정보를 섞는다"는 말을 들으면 W₁, W₂의 shape을 보고 즉시 "아니다"를 판단할 수 있다.
7

이 질문 방식의 실무적 단점

단점 1
개발 속도가 느려질 수 있음

모든 개념을 깊게 파다 보면 간단한 실험 하나도 이론 확인 이후로 미뤄질 수 있다.

대응 전략:
지금 목표가 세미나인지, 구현인지, 최적화인지에 따라 이해 깊이를 다르게 잡는다. "동작하면 된다"가 목표일 때는 Level 1 이해로 충분하다.
단점 2
지금 몰라도 되는 원리에 오래 머무를 수 있음

downstream task를 fine-tuning하는 단계에서 quantization 원리까지 파는 것은 과잉 투자다.

대응 전략:
"이 이해가 현재 task에 어떤 의사결정을 바꾸는가?"를 먼저 물어본다. 의사결정에 영향 없으면 다음으로 넘어간다.
단점 3
팀원 커뮤니케이션이 어려워질 수 있음

"QKᵀ의 row가..."로 설명하면 개념 설명을 원하는 팀원과 소통이 안 될 수 있다.

대응 전략:
청중에 맞게 추상화 수준을 전환하는 연습을 한다. 깊은 이해가 있으면 오히려 쉽게 설명하기도 더 쉽다.
단점 4
제품 관점보다 구조 검증에 과몰입할 수 있음

사용자가 원하는 결과가 나오는지보다 내부 동작이 정확한지에 더 관심이 집중될 수 있다.

대응 전략:
"구조 확인"과 "성능 확인"을 다른 타임박스로 분리한다. PR 기준으로 구조 검증은 PR 내, 제품 검증은 별도 sprint로 나눈다.
단점 5
라이브러리를 믿고 쓸 수 있는 부분까지 직접 확인하려 할 수 있음

torch.nn.MultiheadAttention이 올바르게 구현되어 있는지 소스까지 확인하다 시간을 쓸 수 있다.

대응 전략:
"내가 직접 제어해야 하는가?" 기준을 세운다. 직접 inference loop를 쓰거나 export할 때만 내부를 확인하고, standard usage는 라이브러리를 신뢰한다.
8

실무적으로 가장 좋은 균형

이 자료 전체를 관통하는 이해 방식(tensor shape, mask 위치, gradient 흐름)은 필요하지만, 모든 상황에서 같은 깊이로 파고드는 것은 비효율적이다.

결론
핵심 구조(Level 2)는 tensor/gradient 수준까지 깊게 이해하되, 제품화 단계에서는 "지금 의사결정에 필요한 깊이"까지만 파고 다음 실험으로 넘어가는 것이 좋다.

이 자료가 설명하는 이해 방식의 목표는 "모든 것을 처음부터 구현할 수 있는 능력"이 아니라, "겉핥기 설명을 읽었을 때 연산 그래프 기준으로 그 말이 성립하는지 검증할 수 있는 능력"이다.

이해가 깊이 필요한 시점

  • Shape mismatch 버그가 재현 불가능할 때
  • 논문을 직접 구현해야 할 때
  • Model을 edge device에 export할 때
  • KV cache, quantization 등 최적화 적용 시
  • 이상한 attention pattern이 의심될 때

추상화된 이해로 충분한 시점

  • HuggingFace API로 fine-tuning할 때
  • Prompt engineering, few-shot 실험할 때
  • 팀원에게 Transformer 개요를 설명할 때
  • 이미 검증된 라이브러리 pipeline을 쓸 때
  • 성능 지표(BLEU, perplexity)를 비교할 때
